CheckBox.js


All you need in a checkbox!


The checkbox.js plugin is used to style and manage customized checkboxes on web pages. It does not overide the default checkboxes but only improves the way it is being rendered through the use of custom checkboxes. This plugin comes with 20 plus custom checkbox image pairs with extended support for icons, labels, toggles, animations and events. Documentations are available in the README.md file or you can find full documentation on github.io page.

Installation

Installed by adding the script into the head section of your project file.
Installation locally 
<script src="checkbox/checkBox.js">
Installation cdn link 
https://cdn.jsdelivr.net/gh/teymzz/checkBox/checkBox.js
It is important to note that images and design do not come with javascript. You have to download the full plugin to get images that comes with the plugin. Once the plugin has been installed, we can head on to initialize the plugin.


Plugin Setup

Before initializing the plugin, we need to setup the html structure correctly. This can be done by defining a custom checkbox element followed by the relative native checkbox. This structure ensures that once the plugin is initialized, all native input checkboxes remain hidden leaving only their relative controller custom checkboxes. It also allows the plugin to know which custom checkboxes it should target. A basic sample is shown below:

CheckBox (basic sample) 
<div checkbox></div> // custom checkbox

<input type="checkbox"> // native checkbox

The html structure makes it possible to add flexibility to custom checkboxes through the use of markers. The checkbox plugin supports the use of single or double markers which can be used for custom toggles. The sample html structure of adding markers are shown below:

CheckBox (Single marker) 
<div checkbox>
  <div marker style="background-color: #594fc8;"></div>
</div>
<input type="checkbox">

CheckBox (Double markers) 
<div checkbox>
  <div marker style="background-color: #ffffff;"></div>
  <div marker style="background-color: #594fc8;"></div>
</div>
<input type="checkbox">

Once we define the checkbox similarly to any of the html structures above, we can easily setup the checkbox plugin using any of the two methods below:

Plugin intialization (Method 1) 
<script>
new CheckBox(); //initialize and execute
</script>

By default, the checkbox uses the [checkbox] attribute selector to select all custom checkboxes. The sample script above is the most basic way of initilializing the plugin and at the same time, executing it immediately.

Plugin intialization (Method 2) 
<script>
let checker = new CheckBox(true); //prevent early execution

checker.check(); //execute
</script>

The second initialization method above is used when there is need to call the checkbox multiple times with different configurations which at most times is not necessary.

Plugin configuration

The plugin can be configured by supplying predefined configuration keys and values pairs as argument either when the plugin is intialized or through the check() method. These keys are explained below:

target This is used to change the default custom checkbox [checkbox] selector.
marker This is used to change the default custom checkbox marker [marker] selector.
size This is used to set the default size of the markers, accepting any valid unit of measurement. The string value supplied is used for setting both the width and height of the markers. However, width and height can also be customized by the use of object. Some samples are shown below:
size: setting width and height 
size: '10px' //set width and height
size: setting width and height 
size: {
    width:  '10px', //set width
    height: '30px'  //set height
}
size: setting width and height using axis 
size: {
    x: '10px', //set width
    y: '30px'  //set height
}
The equivalent of this key is the data-size attribute which can be applied specifically on custom checkboxes. This attribute takes a value which optionally can be separated by pipe. The left value is usually the width while the right value is the height but if only one value is supplied, the value will stand for both the width and the height of the custom checkbox. The sample format is shown below :
size: setting width and height using attribute 
<div checkbox data-size="10px|30px">

flip This is takes a boolean value of true or false. This key comes into use when two markers are defined within the checkbox. The true value ensures that both markers are not displayed at once which means that the first marker will only be displayed when the checkbox is unchecked while the second marker will only be displayed when the checkbox is checked. This helps to create an easy toggle effect on the custom markers. The equivalent atribute is the data-flip attribute which can be set with a string value of 'true' or 'false' .

fit This is used for fitting checkbox images. It takes a boolean value of true or false while the default value is true. Assuming the default [marker] selector is not changed, the true value sets the following css on the selected markers.
Fit css 
[checkbox] [marker] {
    background-size: cover;
    background-position: center top;
    background-repeat: no-repeat;
}
The the sample css above, is generated dynamically and added to the page based on the target [checkbox] and the target [marker] selectors used. This css may change if any of the selectors were redefined.

init This takes a callback function and is mostly used to set up a function that must be called when the checkbox is initialized. It provides an argument which is an object that makes it easier control the entire custom checkbox. These argument object contains keys which are native, custom, marker, flip, fit, checked and value . The example below shows how the init() function can be used.
init sample 
<script>
new CheckBox({
    init: function(checkbox) {
        
        let value = checkbox.value;

        if( checkbox.checked ) {
            console.log(`checkbox ${value} is: checked`)
        } else {
            console.log(`checkbox ${value} is: unchecked`)
        }

    }
});
</script>

In the sample above, the checkbox arguments is an object that returns each of the listed methods below:

native : This refers to the native input checkbox (hidden).
custom : This refers to the custom relative checkbox (visible).
marker : This refers to the markers within the custom checkbox.
flip : This returns true or false based on the default flip configuration value.
fit : This returns true or falsetrue or false based the default fit configuration value.
checked : This returns true or false based on the status of the native checkbox.
value : This returns the current value of the native checkbox when it is checked.

toggle This method defines a callback function that must be called every time a checkbox is toggled. The argument it takes returns an object which has keys similarly to the init() method with some extra keys which are the color, label, init keys.
The added init key is a special method that is assigned for accessing the init() callback method easily within the toggle() method. The example below shows how this method can be used.
sample: toggle 
<script>
new CheckBox({
    init: function(checkbox) {
        
        let value = checkbox.value;

        if( checkbox.checked ) {
            console.log(`checkbox ${value} is: checked`)
        } else {
            console.log(`checkbox ${value} is: unchecked`)
        }

    },
    toggle: function(checkbox) {
        
        checkbox.init();

    }
});
</script>

The sample code above is one which defines that the checkbox should run the init() method immediately the class is initialized. The toggle() function will ensure that the init() method is always executed whenever the checkbox is toggled.

The data-func attribute is used to define a callback function name that is called when the checkbox is toggled. This function is never called immediately the plugin is loaded unless an attribute of data-init is added to the custom checkbox.

assign This key takes an object as its value. The basic function of this is to assign the value of an attribute to another attribute based on the status of the checkbox. The format resembles the structure below :
assign format 
assign: {
    attr: ['source', 'destination', 'unchecked|checked|both|:marker1|:marker2|:markers'],
    alter: (source_values) => {
        return source_values
    }
}
assign html sample format 
<div checkbox source="foo" destination=""></div>
<input type="checkbox">
The assign key has two subkeys attr and alter. These keys are used to manage the toggling of custom box attributes' values.

The attr key is used to toggle the value of one attribute (e.g source) to another attribute (e.g destination). The source must be an attribute on the custom checkbox whose value is pushed or pulled from the destination when the checkbox is checked or unchecked respectively. Both the source and destination attributes can be of any valid attribute name.

The third array value of attr is a list of options that specify the state of the checkbox at which a pushing of value from source to destination should occur. The inverse of this specified state will then automatically become when the pulling event will occur. For example, pushing of values from one attribute to another could be when the checkbox is "checked", "unchecked" or in "both" cases depending on the option supplied. If the pushing is done when checkbox is checked, then the pulling back (or removal) of the value from the destination attribute (where it was added) will occur when it is unchecked. If the value "both" is used, this means that the attribute should be included in both cases. The :marker options are only used in cases where the attributes are expected to be added to the first marker ":marker1" or the second marker ":marker2" or both markers ":markers".

attr sample 
assign: {
    attr: ['data-names', 'class', 'checked'],
}
attr html sample format 
<div checkbox data-names="foo" class=""></div>
<input type="checkbox">

In the sample structure above, the 'data-names' is expected to be an attribute on the custom checkbox whose value will be added (or pushed) into the 'class' attribute of the same custom checkbox. This value will only be added when the checkbox is 'checked' and it will be removed (or pulled) when the checkbox is unchecked. Note that the default option for the third argument is 'checked' if not specified.

The alter key is used to modify the value of the source attribute's values. It works along with the attr key. The source attribute's value are separated into arrays that can be easily modified before they are added to the destination attribute. This is mostly useful for attributes that may contain multiple values.

alter sample 
assign: {
    attr: ['data-names', 'class', 'checked'],
    alter: (dataNames) => {
        return dataNames.map( el => 'name_' + el );
    }
}
alter html sample format 
<div checkbox data-names="foo bar"></div>
<input type="checkbox" value="A">

In the sample format above, the values obtained from data-names will be accessed and modified by the alter(). Each of the values will be assigned a prefix of 'name_'. Hence we would have 'name_foo name_bar' added into the class attribute of the custom checkbox when it is checked. This value will also be removed when the checkbox is unchecked. In this way, it becomes easier to modify attributes. In a situation where we desire to toggle the data-names the third attr key's array option can be defined as ':marker1' option that is used for when checkbox is checked or ':marker2' used for when checkbox is unchecked or ':markers' which is used for both cases.

The data-assign attribute is similar to the assign key. Separated by pipe, the first value defines the animation stage, the middle value specifies the source attribute while the last part specifies the destination attribute. The sample format is shown below:
data-assign html sample format 
<div checkbox data-assign="checked|source|destination" source="foo" destination=""></div>
<input type="checkbox">

In the sample above, we don't necessarily need to define the entire options for data-assign attribute. We can either define a single option of the animation stage, that is, "checked", "unchecked", "both", ":marker1", ":marker2", ":markers", or we can define the entire argument specifying both the source attribute along with the destination attribute.

Sample: Script

Script 
<script>
new CheckBox({
    toggle: function(checkbox) {
        let custom, marker, value, animate, anime = [];

        custom = checkbox.custom;
        marker = checkbox.marker;
        value  = checkbox.value;

        animate = custom.getAttribute('data-anime');

        if( animate ){
            anime = animate.split(' ');
            anime.unshift('animated');
            anime = anime.map( el => 'animate__' + el );
        }

        if( checkbox.checked ) {

            if(animate) marker[1].classList.add(...anime);
            console.log(`checkbox ${value} is: checked`)
        
        } else {

            if(animate) marker[1].classList.remove(...anime);
            console.log(`checkbox ${value} is: unchecked`)
        
        }

    },
    flip: true,
});
</script>

It is amazing that only one custom script was used to handle the entire custom checkbox samples smartly. The format of our html structure already ensures that each custom box is well associated with its native checkbox. Through the use of the checkbox custom script below which was defined earlier, we were able to control all checkboxes.

Previous Script 
<script>
new CheckBox({
    toggle: function(checkbox) {
        let custom, marker, value, animate, anime = [];

        custom = checkbox.custom;
        marker = checkbox.marker;
        value  = checkbox.value;

        animate = custom.getAttribute('data-anime');

            anime.unshift('animated');
            anime = anime.map( el => 'animate__' + el );
        }

        if( checkbox.checked ) {

            if(animate) marker[1].classList.add(...anime);
            console.log(`checkbox ${value} is: checked`)
        
        } else {

            if(animate) marker[1].classList.remove(...anime);
            console.log(`checkbox ${value} is: unchecked`)
        
        }

    },
    flip: true,
});
</script>

CheckBox smart script

The previous script can be re-written smartly by taking advantage of the attribute toggling system that is integrated to the plugin through the "assign" key. This is shown below:

Script 
<script>
new CheckBox({
    assign: {
       attr: ['data-anime', 'class'],
       alter: (dataAnime) => {
          dataAnime.unshift('animated');      
          return dataAnime.map(el => 'animate__' + el);
       }
    },
    toggle: function(checkbox) {
        
        let value = checkbox.value;

        if( checkbox.checked ) {
            console.log(`checkbox ${value} is: checked`)
        } else {
            console.log(`checkbox ${value} is: unchecked`)
        }

    },
    flip: true,
});
</script>

In the sample above, the "assign" key uses its sub key "attr" to set two attributes on the custom boxes in which the first attribute contains a value that should be added to the second attribute if the checkbox is checked. The beauty of the "attr" key in the script above is that it makes it possible to shift the animation type using a third array value. Since the default value for the third array value is "checked" we do not need to redefine this option unless we need to use a different option. An option of "unchecked" will change this behaviour and ensure that the value is added only when the checkbox is unchecked and it will be removed back once it is checked. The sample script for when it is "unchecked" is similar to the script above. This is shown below:

Script 
<script>
new CheckBox({
    assign: {
       attr: ['data-anime', 'class', 'unchecked'],
       alter: (dataAnime) => {
          dataAnime.unshift('animated');      
          return dataAnime.map(el => 'animate__' + el);
       }
    },
    toggle: function(checkbox) {

        let value = checkbox.value;
        
        if( checkbox.checked ) {
            console.log(`checkbox ${value} is: checked`)
        } else {
            console.log(`checkbox ${value} is: unchecked`)
        }

    },
    flip: true,
});
</script>

In the script above, the attributes' values will be pushed only when the checkbox is unchecked. One other thing to note is that animations may not trigger immediately the page loads unless it is set as "both". If we need the animation to trigger as earlier, this means that we need to push the attribute immediately the page loads, we need to set the init key as a callback or as a boolean with the value of true. The script below is a sample of how we can help animations to trigger earlier if the default state of the checkbox is unchecked.

Script 
<script>
new CheckBox({
    assign: {
       init: true,
       attr: ['data-anime', 'class', 'unchecked'],
       alter: (dataAnime) => {
          dataAnime.unshift('animated');      
          return dataAnime.map(el => 'animate__' + el);
       }
    },
    toggle: function(checkbox) {

        let value = checkbox.value;
        
        if( checkbox.checked ) {
            console.log(`checkbox ${value} is: checked`)
        } else {
            console.log(`checkbox ${value} is: unchecked`)
        }

    },
    flip: true,
});
</script>

Using Labels

Labels are texts that immediately follows the input checkbox fields. In order to add labels, the custom checkbox is assigned with an attribute of data-label which contains the text label itself that is either single or separated with pipe. When the text is separated by pipe, the left side of the text will be for when the checkbox is unchecked while the right side of the text will stand for the displayed label when it is checked. Once this attribute is added, the label field itself should also be assigned with an attribute of data-access="label" which will help the checkbox to know where to add the text label. Cache Fixes

Cache fixes are necessary when checkboxes updates their default state after user interaction due to cache storage. For example, a checkbox that was not checked by default may end up being checked when reloaded due to stored user interaction. In the event that we need to ensure that the checkbox is always unchecked when a page reloads, we need to add the attribute checked on the native checkbox and the value must be set as "false". All native checkboxes defined this way will be unchecked by default even if the last stored user interaction was checked. Similarly, we can set the checked value as "true" which will ensure that the checkbox is always on by default regardless of user interaction.


In the previous password input box example, we needed to add the attribute data-init to the custom checkbox to ensure that if the checkbox was checked by default, then the function can run. Running a callback function earlier as in the case of password field may end up exposing the user details once the page loads if the last user interaction stored was that the checkbox was checked. In order to fix this issue, a better option will be to ensure that the password field is never checked as soon as the page loads. In order to do this, we can redefine our html code as shown below :


checkbox L (cache fix) 
<div class="checkers box">
  <div>checkbox L (cache fix)</div>    
  <div>
    <div class="in-flex wrap rad-r" style="align-items: center; gap: .2em">
      <div checker class="in-flex" data-init data-func="view2" data-color="@red|lime" data-anime="fadeIn" data-assign=":markers">
      
        <div marker class="red">
          <span class="flex fit mid-v">
            <span class="flex bi bi-eye-slash"></span>
          </span>
        </div>
      
        <div marker class="lime">
          <span class="flex fit mid-v">
            <span class="flex bi bi-eye"></span>
          </span>
        </div>
        
      </div>
      <input type="checkbox" checked="false" value="L">
    </div>
    <div>
      <input id="pass2" type="password" class="rad-r">
    </div>
  </div>
</div>
Just to prevent conflicts, in the above, the new password field id attribute is set as "pass2", while the callback function data-func is renamed to "view2". The new callback function will resemble the structure below:
New Script 
<script>
function view2(checkbox) {

  let input;
  
  input = document.getElementById('pass2')
  
  if( checkbox.checked ) {
  
    input.setAttribute('type', 'text')
    
  } else {
  
    input.setAttribute('type', 'password')
  
  }
  
}
</script>
Now, since the checked attribute is set to "false" on the native checkbox, the checkbox will always maintain an unchecked state when the page reloads.

Base Reversal

The checkbox "reverse" effect is one in which is simalar to the detachment of a checkbox from the list of toggeled boxes. In this case, rather than a complete detachment, any native checkbox with an attribute of data-bind having a value of "reverse" will alway negate the parent checkbox. An example is shown below:

In the sample above, the relationship between the third and fourth checkbox along with the parent checkbox is that they are always the opposite of the parent checkbox which is not the case for the third checkbox that is a direct reflection of the parent. The fifth checkbox is however completely detached from this relationship. A more complex relationship is shown below:
checkbox U2 (Base Reversal)
  (parent)

  (no data-bind)
  (no data-bind)
  data-bind = 'free'
The example above is similar to the first one. Although, it is true that the second and third always negate the parent checkbox. In this situation, that may not be entirely true and this is because of the effect of the two checkboxes (third and fourth) that have direct relationship with the parent. The parent element is only allowed to be toggled on when both of the children checkboxes that neither negates it nor freely exist within its group are toggled on. This means that because the third and fourth are children that agree, the parent element will have to wait until both of these children are toggled on before finally being toggled on. The reversed checkboxes will wait till the parent and both of the agreeing children are altogther in a single accord before finally negating them.
